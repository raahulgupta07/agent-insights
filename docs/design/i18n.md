# Internationalization (i18n)

This document describes how Dash supports multiple languages and locales in the frontend, what contracts the backend exposes, and the conventions contributors should follow when adding new UI strings.

## Scope

- Supported UI locales: `en` (English, default), `es` (Spanish), `he` (Hebrew, RTL).
- Routes are **not** localized — URLs do not carry a locale prefix. The active locale is user preference, stored in `localStorage` and optionally hydrated from the backend org config.
- Current translation catalogs live under `/locales/{en,es,he}.json` at the repository root (imported by both the Nuxt app and tests).

## Stack

| Piece            | Choice                                              |
| ---------------- | --------------------------------------------------- |
| Library          | `vue-i18n@9`, composition mode (`legacy: false`)    |
| Loader           | Static JSON import in `frontend/plugins/i18n.ts`    |
| Interpolation    | Named parameters: `t('key', { name })`              |
| Inline markup    | `<i18n-t keypath="..." tag="span">` with slots       |
| RTL              | `document.documentElement.setAttribute('dir', ...)` |
| Formatting       | `Intl.DateTimeFormat` / `Intl.NumberFormat` presets |

## Plugin contract

`frontend/plugins/i18n.ts` registers a single global `createI18n` instance.

- Persists the user-selected locale under the `dash.locale` `localStorage` key.
- Falls back to `en` whenever a stored or backend value is not a known locale.
- Provides a global `$setLocale` (via `nuxtApp.provide('setLocale', ...)`) which updates the active locale, persists it, and syncs `<html lang>` + `<html dir>`.
- Hydrates once on boot from `GET /api/config/i18n`. The backend response may include `current_locale` — this is honored only when no user-selected value exists in `localStorage`.
- Registers three `Intl` formats: `short`/`long` (dates), `decimal`/`integer`/`percent`/`currencyUSD` (numbers).

RTL locales are whitelisted in the plugin (`he`, `ar`, `fa`, `ur`). The Tailwind CSS layer is already RTL-safe — components use logical properties (`ms-*`, `me-*`, `ps-*`, `pe-*`, `start-*`, `end-*`) rather than directional ones.

## Catalog shape

Catalogs are nested JSON objects keyed by feature area:

```
{
  "common":   { "save": "Save", "cancel": "Cancel", ... },
  "nav":      { ... },
  "auth":     { ... },
  "settings": {
    "membersTabs": { ... },
    "audit":       { ... },
    "integrations":{ ... },
    "aiSettingsPage": { ... }
  },
  "evals":    { "run": { ... }, "pagination": { ... }, ... },
  "monitoring": { "diagnosis": { ... } },
  "queries":  { "detail": { ... }, ... },
  ...
}
```

All three locale files must have identical shape. The ordering in `en.json` is authoritative — translation files preserve key insertion order via `json.load(..., object_pairs_hook=OrderedDict)` in helper scripts.

## Authoring rules

1. **Every user-facing string MUST come from the catalog.** No hard-coded literals in templates or script-side UI output (toasts, `confirm()` prompts, `alert()` bodies).
2. **Template**: prefer `$t('ns.key')` / `$t('ns.key', { name })`. For strings with embedded markup, use the `<i18n-t>` component with slots (see `settings.aiSettingsPage.llmImpactInspect` for an example).
3. **Script (`<script setup>`)**: grab the helper once at the top of the block:
   ```ts
   const { t, locale } = useI18n()
   ```
   Use `t(...)` for any user-visible string returned by a function (toasts, formatters, dialog text). Use `locale.value` when passing to `Intl.*` APIs.
4. **Params, not concatenation.** Prefer `t('audit.rangeCount', { start, end, total })` over building strings by `+`, since word order varies by language.
5. **Locale-reactive maps.** Any `const MAP = { ... }` that holds localized strings must be wrapped in `computed(() => ({ ... }))` so it re-renders when the locale changes. Lookups become `MAP.value[key]`.
6. **Canonical keys for CSS.** If you have a map from labels to CSS classes, key it on canonical (non-localized) identifiers like `'tool:create_data'`, never on the English display label — otherwise class lookups break in other locales.
7. **Error codes over error strings.** When the backend reports a failure via the error-code catalog (`error_code` field), the frontend maps the code to `t('errors.<code>')` rather than echoing the server string. See `frontend/composables/useErrorMessage.ts`.
8. **Dates and numbers** always go through `date.toLocaleDateString(locale.value, ...)` / `Intl.NumberFormat(locale.value, ...)`. Never hand-format month names or decimal separators.
9. **Plurals / short forms.** We use explicit keys (`queries.showingPublishedOne`, `queries.showingPublishedMany`) with a ternary in the template rather than ICU plural syntax — simpler, covers every language we actively ship.
10. **Relative time.** Call the shared helpers (for example `queries.timeMinutesAgo` / `queries.timeHoursAgo` / `queries.timeDaysAgo`) with `{ n }`. Don't inline `${n}m ago`.

## Adding a new string

1. Add the key to `locales/en.json` under the right namespace.
2. Add translations to `locales/es.json` and `locales/he.json` at the **same path**.
3. Run the sync check (see below) — a CI lint should fail if any catalog is missing or has extra keys.
4. Use `$t('ns.key')` / `t('ns.key')` in the component.

### Sync check

A Python helper compares keys across locales:

```python
import json
def flatten(d, prefix=''):
    out = set()
    for k, v in d.items():
        full = f'{prefix}.{k}' if prefix else k
        if isinstance(v, dict):
            out |= flatten(v, full)
        else:
            out.add(full)
    return out

locs = {}
for loc in ('en', 'es', 'he'):
    with open(f'locales/{loc}.json') as f:
        locs[loc] = flatten(json.load(f))

assert locs['en'] == locs['es'] == locs['he']
```

A stricter tooling option is [`eslint-plugin-vue-i18n`](https://eslint-plugin-vue-i18n.intlify.dev/), which additionally warns on `no-missing-keys`, `no-unused-keys`, `no-raw-text`, and structural divergence. Enable it in a future iteration by adding the plugin to `frontend/` devDependencies and wiring `frontend/.eslintrc.cjs`:

```js
module.exports = {
  extends: ['plugin:@intlify/vue-i18n/recommended'],
  settings: {
    'vue-i18n': {
      localeDir: '../locales/*.{json}',
      messageSyntaxVersion: '^9.0.0',
    },
  },
  rules: {
    '@intlify/vue-i18n/no-missing-keys': 'error',
    '@intlify/vue-i18n/no-unused-keys': ['warn', { extensions: ['.vue', '.ts'] }],
    '@intlify/vue-i18n/no-raw-text': ['warn', { ignorePattern: '^[\\s—–·•→·:]+$' }],
  },
}
```

## Backend contracts

- **Config**: `GET /api/config/i18n` returns `{ available_locales: string[], current_locale: string | null }`. The frontend uses `current_locale` as the boot default only when the user has not chosen explicitly.
- **Error codes**: when a request fails, the backend returns an `{ error_code, message, params }` body (alongside `status_code`). The frontend resolves `error_code` against the `errors.*` namespace; `message` is an English fallback for logs only. See the backend `AppError` module.
- **Emails**: notification email templates live under `backend/templates/emails/{en,es,he}/*.jinja2`. The backend picks the recipient's locale (user > org > `en`), renders the template, and sets `dir="rtl"` on the `<html>` tag for RTL languages.

## RTL checklist

- All custom CSS uses logical properties (`ms-*`, `me-*`, `ps-*`, `pe-*`, `start-*`, `end-*`) — no `ml-*`, `mr-*`, `pl-*`, `pr-*`, `left-*`, `right-*`.
- Icons that encode direction (arrows, chevrons for navigation) mirror automatically via `dir="rtl"` on the root; ad-hoc SVGs should use `rtl:-scale-x-100` when they carry directional meaning.
- Numeric sequences, code blocks, and monospaced identifiers are wrapped in `<bdi>` when they appear inline with RTL text to preserve bidirectional rendering.

## Testing locally

```bash
# Frontend dev server
cd frontend && yarn dev

# Switch locale from the UI language switcher, or programmatically:
localStorage.setItem('dash.locale', 'he'); location.reload()
```

Playwright smoke tests run against all three locales. A test marked `@rtl` asserts that `document.documentElement.dir === 'rtl'` before interacting with navigation.
