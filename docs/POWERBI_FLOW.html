Per-user sign-in → discover → gate → ground → answer. Every step is per-user (own account, own data, own cache).

Lane A — Connect & Onboard (once per user, per account)
1

Sign in Microsoft

User enters their own email + password (ROPC). No admin / service account. MFA → falls back to device-code sign-in.

Token scope: Dataset.Read.All · identity captured as ms_account_email. Agent is named after this account.
2

Discover — scan ALL reachable workspaces via their reports Microsoft

Enumerate the user's reports everywhere they have access, then follow each report back to its dataset (semantic model).

My Workspace/reports
Every App/apps → /apps/{id}/reports
Every Workspace/groups → /groups/{id}/reports
Report-driven (a shared user's raw /datasets list is often empty). Each report carries datasetId + workspaceId → datasets deduped.
3

Gate each dataset — can this user QUERY it? Microsoft

Probe every discovered dataset (_probe_queryable):

✔ QueryableRead + dataset access → keep & enumerate
◐ View-onlyreport access, no dataset access → "needs Build permission", surfaced but not queried
✕ No accesshidden
This is the read-vs-query gate. Report-view alone ≠ permission to query the data.
4

Enumerate tables, columns, types Microsoft

For each queryable dataset, read the real schema via INFO.VIEW.TABLES() + INFO.VIEW.COLUMNS() — actual names + data types (no guessing).

5

Persist & seed the agent

Store an offline index (table name → datasetId / workspaceId) + seed the agent's table rows. Later queries resolve the dataset offline — no live re-scan.

6

Relevance classify — keep business, hide noise v1.78.7

Classifier is authoritative: activates business tables (facts, dimensions) and deactivates noise (usage-telemetry, staging copies, measure-holders).

Proven: 10 business tables kept, 15 telemetry/staging hidden. (Fixes the "new agent = 0 tables" skeleton.)
7

Learn from data — sample real values v1.76

Sample a few real rows per active table (PII-safe) → example column values feed the onboarding AI, so it describes your actual domain, not a guess from the connector name.

8

Fetch measures + relationships v1.78

INFO.VIEW.MEASURES() → the model's pre-built KPIs · INFO.VIEW.RELATIONSHIPS() → the join graph. Both stored for grounding.

Example (Project Tracking): 17 measures + subjects[project_id] → projects[id] relationship.
9

AI onboarding AI

Generate the agent's description + conversation starters + primary instruction — grounded in the real tables, sampled values, measures & relationships.

10

Hot Start — make the data "hot" on open v1.79

On agent open, in the background: warm the engine (fire a cheap query per dataset so Microsoft loads the model into memory) + pre-compute headline KPIs from the model's measures (cached per-user).

Lane B — Ask a question (every query)
Q1

User asks in natural language

e.g. "how many projects are on risk?" · Planner routes it (≥ balanced model for Power BI; answers simple asks directly, no double-clarify).

Q2

Build the user's client (per-user) isolated

Constructs a client with the user's own credentials + the offline index + measures/relationships. Sets _bow_user_id so the result cache is keyed per-user (RLS-safe).

Q3

AI writes the query AI

Generates a Python function that runs DAX. Grounded rules: use bare table names, prefer measures, join related tables in ONE DAX query (never a pandas merge → no cartesian).

Q4

Execute — with safety nets Microsoft

Before running: strip any Dataset/Table qualifier → bare name. Check the per-user cache.

Cache hit< 1ms (warm / repeat)
Cache miss→ POST /executeQueries · MS engine 40–84s cold, fast when warm
200 + errorrow/table-limit caught here too
✔

Success answer

DataFrame → answer + optional chart. Cached per-user for repeats.

!

Failure → clean message no raw errors

Every error is classified & translated (raw JSON logged server-side only):

no_access / auth → "You can view this report but can't query its data — ask for access." (stops, no retry)
not_found → "couldn't find that table…"
invalid_dax → self-correct & retry (measures + relationships in hand)
too_much_data → "I'll summarise it"
throttled (429) → auto-retry, silent
Per-user isolation everywhere: own account · own token · only the datasets their account can query · overview, headline & DAX-result caches all keyed by (data_source, user_id) → a Row-Level-Security dataset never serves one user's rows to another.
Hard limits (Microsoft-side, cannot fix):
• 40–84s cold query (engine load) · warm = fast
• 1 result table per query · 100k rows / 1M values / 15MB · 120 queries/min
• Build/dataset access required — view-only can't be queried
Open items:
• Never-synced agent shows a skeleton (should prompt "Sync now")
• Reconnecting can spawn duplicate empty agents
• Optional: exhaustive per-workspace dataset scan (catch datasets with no visible report)
