The one insight: every learning store today is keyed data_source_id = the private clone = per user. So learning never compounds across users who share a semantic model. Fix = a second store keyed by dataset_id (the PBI semantic-model GUID) — identical only for users with the same access, so sharing is safe by construction.

4 · Two-plane learning (what to build)

SHARED plane — model_knowledge

Keyed dataset_id (NOT user). Org-level. Only schema-shaped logic.

  • meaning — table/col → business sense
  • join — canonical relationship graph
  • dax_template — parametric DAX, {placeholder} only
  • mistake — failed_dax → error → fixed_dax (never-repeat)
  • verified_count++ when 2nd user confirms

PRIVATE plane — clone stores

Keyed data_source_id. Per user. Never shared.

  • answer_cache — actual result rows
  • headline KPIs — computed values
  • brain_graph_edge — entity values
  • anything containing a literal data value
🔒 LEAK FIREWALL (on promote private → shared): sanitizer strips every literal row value, WHERE-clause constant, entity name, ID, date, PII. Ambiguous → drop. Only table/column names + query shape + business-meaning text cross. A DAX template is safe; a cached result is not.

5 · Access gate — how sharing stays isolated

RETRIEVE for U (build context): models_U = { dataset_id for each table in U's clone } # from ConnectionTable.metadata_json.powerbi.datasetId knowledge = SELECT * FROM model_knowledge WHERE dataset_id IN models_U # <-- intersection = the whole ACL inject(knowledge) alongside U's private clone overview WRITE (capture, step ⑧): learning = extract(query, result) safe = sanitize(learning) # leak firewall UPSERT model_knowledge(dataset_id, kind, content=safe) ON CONFLICT verified_count += 1
Because retrieve filters dataset_id IN (models this user's clone holds), a user physically cannot receive knowledge for a model absent from their own catalog. Access-scoping falls out of the same GUID — no separate permission table.
