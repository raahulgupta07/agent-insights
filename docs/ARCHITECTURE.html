0Goals & principles

One codebase, one process, one deploy. dash's patterns and the brain are ported natively into bagofwords (bow). Agno is reference-only; OpenRouter is the single LLM gateway — so there is no SDK/dependency clash.

  • bow is the chassis — Nuxt UI, multi-tenant/RBAC, 46 connectors, observability, MCP, Instructions + approval gate, scheduler. Kept whole.
  • dash is a pattern, not a service — dual-schema, Engineer view-builder, DB-level read-only, learning loop reimplemented in bow idioms.
  • Everything learned is gated — memories, shared skills, correlations land in pending, go live only after approval (reuse bow's Instruction build/approval).
  • The loop is the exception — most traffic served from cache + precomputed views; the LLM agent loop serves only the novel tail.
  • Every user authors skills — personal = instant & safe-by-construction; sharing = gated. Authoring is "do it once, save it."
  • Correlate anything — DuckDB federates live + cached + files; the brain holds the learned cross-source key map.
De-risked: the one thing that could kill the merge — does bow's native tool_use planner run through OpenRouter? — is verified. openai_client.py:235 implements inference_stream_v2 over standard Chat Completions streaming tool_calls. The custom provider points it at openrouter.ai/api/v1. The core loop works.

1Master architecture

Seven bands top-to-bottom + two cross-cuts (observability, infra). A question and incoming data both traverse the stack.

Band 1 · Interfaces

How 1000 users reach it

  • Nuxt UI · MCP clients (Cursor/Claude) · Slack/Teams · REST/SDK · scheduled tasks
Band 2 · LLM gateway

OpenRouter — one key, any model

  • bow custom provider → base_url=openrouter.ai/api/v1. Strong model for analysis, cheap model for routing/classify. Native tool_use over Chat Completions.
Band 3 · Control plane (bow)

The platform

  • orgs · RBAC · Connections · Reports/Dashboards · per-org quota · auth (JWT / API-key / OAuth2-PKCE)
Band 4 · Agent core — AgentV2 plan→execute→reflect

The brain's hands

  • Tools registry (+ Analyst, + Engineer build_data_asset)
  • Context Hub builders — schema · instructions · query-patterns · observations · brain · skills
  • Code-exec — SQL + python, AST guard + DB-level read-only engine
Band 5 · Skills (self-service)

Every user authors capabilities

  • SKILL.md packs · 3 levels (catalog → load_skill → bundled files) · scope: personal / org / global · promote-from-chat · top-K retrieval per user
Band 6 · Karpathy 2nd-Brain

Long-term memory + self-improvement

  • Instructions (approval gate) · learned memories · reasoning-cache · insights · entity/correlation graph
  • writes: 👎 self-distill · harness mining · proven-SQL · insight daemon — all → pending → active; daemons leader-gated
Band 7 · Data

Storage + federation

  • OLTP Postgres (app) — psycopg_pool · OLAP Postgres+pgvector(+AGE): public(RO) · analytics · staging
  • DuckDB federation · MinIO/S3 Parquet+files · Redis answer-cache+qbank · 46 connectors

2LLM gateway — OpenRouter

One provider, any model, one key. Removes all per-SDK dependency conflict (no Agno, no openai 2.x).

ItemSetting
provider typecustom (openai-compatible) in bow's LLM layer
base_urlhttps://openrouter.ai/api/v1
keyOPENROUTER_API_KEY → Bearer header
client pathopenai_client.py → inference_stream_v2 (Chat Completions streaming tool_calls) ✓ verified
default modeltool-capable (e.g. anthropic/claude-* or openai/gpt-*) — planner emits native tool_use
router modelcheap model for classify/route/title (cost control)
not available via OpenRouterAnthropic extended-thinking, OpenAI-Responses web_search (provider-native) — optional, not the loop
Pin tool-capable models. Not every OpenRouter model supports tool calling; a wrong default silently breaks the planner loop.

3Serving funnel — the scale lever

At 1000 users the design wins by serving most traffic where there is no LLM call.

~70% · 0 LLM · ~1-2s① Redis answer-cache — known question, org-scoped → return directly
0 LLM · live numbers② Reasoning-cache (param-swap) — nearest proven SQL → swap literals → re-run live
0 LLM · fast SQL③ Materialized analytics.* view — Engineer precomputed it on schedule
~30% · LLM-bound · semaphore④ Agent loop — novel question only → plan/execute/reflect via OpenRouter
Biggest unproven assumption: the ~70% hit-rate. New product = cold caches. Measure hit-rate on real traffic before trusting the 1000-user economics. The funnel is also the cost funnel.

4Flow A — a question

  1. User asks → gateway authenticates, resolves org_id+user_id, checks org+user quota.
  2. ① Redis answer-cache. HIT → return + SOURCES. Most users stop here.
  3. MISS → ② reasoning-cache: embed → nearest proven SQL → param-swap → re-run live, 0 LLM.
  4. MISS → ③ materialized view? → plain fast SQL.
  5. Novel → ④ agent loop. ContextHub primes: schema + instructions + skill catalog (top-K, user-scoped) + query-patterns + brain memories + correlation-map.
  6. Planner (OpenRouter, via semaphore) decides. Domain match → load_skill injects SKILL.md + scopes its tools.
  7. Tool exec: read via read-only engine (public live) or DuckDB cross-source; Engineer build_data_asset writes analytics.*.
  8. Observation → refresh_warm → loop until done / circuit-break. Harness mines proven SQL → brain pending.
  9. Stream answer (SSE) → Nuxt renders. Full trace written.
  10. Write-back: cache ①, store proven SQL ②. 👎 → self-distill → pending memory. "Save as skill?" offered.

5Flow B — data in

  1. Admin connects a source (bow Connection): creds → Fernet-encrypted → app DB.
  2. Schema introspect → DataSourceTable catalog + stats (centrality/richness).
  3. Freshness per asset: live cached(TTL) materialized.
  4. Unstructured (docs/images): extract → embed → pgvector; entities → AGE graph.
  5. Correlation: agent discovers candidate join keys → proposes → admin approves → brain entity graph.
  6. Engineer materializes correlated analytics.* view from approved key-map.
  7. Scheduled refresh (leader-gated): re-snapshot, drift-guard + backup-before-replace, notify on drift.
  8. Analyst & Tier-3 prefer correlated view; common questions precompute into Tier 0/1 cache.

6Skills — self-service, every user authors

A Skill = packaged capability (procedural SKILL.md + bundled assets), loaded on demand. "Like Claude" = a convention + progressive disclosure + a load tool + a code sandbox — all of which bow already has.

Progressive disclosure — 3 levels

L1  always in context (~30 tok/skill)   → SkillContextBuilder injects "name: description"
                                          (top-K relevant, scoped to THIS user's visible set)
L2  on demand                           → load_skill(name) → full SKILL.md body injected
                                          + skill's required_tools added to the run
L3  deep, read/run only if needed       → bundled queries.sql / helpers.py / build.sql
                                          via read_file · AST sandbox · Engineer

Authoring paths — how a non-coder makes one

  • A · Promote-from-chat (primary). After a good analysis, click "Save as skill." The agent distills the conversation trace — proven SQL, steps, question pattern — into a SKILL.md draft (description auto-written, procedure from trace, queries.sql from executed SQL, required views detected). → personal skill in seconds, no code.
  • B · Plain-language wizard. name · "use when…" · steps in English · pick tables. Agent fills the SQL.
  • C · Raw editor. Monaco SKILL.md (power users; bow ships Monaco).

Promotion ladder

analysis → "Save as skill" → PERSONAL (instant, private)
  → proves useful → "Share to org" → REVIEW GATE (approval + lint) → ORG-SHARED
     → broadly valuable → admin promotes → GLOBAL
Scale gotcha: dump-all descriptions into L1 only works for <~50 skills. With hundreds, embed descriptions (pgvector) and inject only top-K relevant, user-scoped. Same engine as reasoning-cache.
Safety: personal skills are safe-by-construction (DB read-only engine + AST sandbox + own org scope). Only sharing is gated — personal creation stays open so all 1000 users can author freely.

8Federation & correlation

You cannot SQL-join Snowflake ⨝ Mongo ⨝ uploaded-CSV remotely. DuckDB reads them all in one place; Postgres stays system-of-record.

Two-way query = per-asset freshness policy

ModeWhenHow
livesmall, needs-fresh, fast sourcepassthrough each time
cached(TTL)repeat reads, tolerate lagParquet snapshot in MinIO, refresh on TTL
materializedcorrelation, heavy joins, offlineingest → staging; Engineer view; scheduled refresh

Correlation is learned + gated, never hardcoded

  • Brain stores entity definitions + cross-source key maps (salesforce.account_id ⇄ pg.customer_id ⇄ mongo.org).
  • Agent discovers candidate keys → proposes → admin approves → Engineer materializes correlated view.
  • Unstructured: embed (pgvector) + entity-extract into same graph → email entities correlate with structured rows.
Governance: caching external data locally crosses tenant/PII lines. Gate ingestion, encrypt (Fernet), per-org isolation, log every materialize.
